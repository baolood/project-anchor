# PROJECT_ANCHOR_EARLY_DEMO_SCRIPT_V1

本文档定义 **约 20 分钟** 的 Early Demo 口述与走屏流程，面向 **3–5 名真实 crypto 交易者**（或同等经验的评审）。  
边界以 `PROJECT_ANCHOR_EARLY_DEMO_SCOPE_V1.md` 与只读演示方案为准：**不执行真实交易、不接资金、不接真实 API key、不承诺收益**。

---

## 0. 演示目标（一句话）

让参与者理解：Project Anchor 如何把 **纪律执行、风控检查、命令流、运维状态** 串成可观察的系统；当前阶段是 **Early Demo**，展示能力边界而非托管交易或收益工具。

---

## 1. 事前准备（演示前 24 小时内完成即可）

| 项 | 说明 |
|----|------|
| 环境 | 可访问已部署的 **anchor-console**（staging / VPN / 受控 URL）；浏览器无痕或干净配置 |
| 入口 | 确认 `/demo` 可打开；确认 `/ops`、`/commands` 在只读或可控模式下可用 |
| 屏幕 | 关闭消息通知；桌面无真实 API key、无交易所账户密码、无持仓敏感信息 |
| 话术 | 主讲人读过本文档与 Scope §4 **禁止话术** |
| 备案 | 若 live 环境异常：准备 **截图或短视频** 替代 `/ops`、`/commands` 实时画面 |

**对外场子硬规则：** 打开 `/commands`（控制台首页）时 **只做滚动与讲解**，**不点击** Create、Retry 等任何写操作；需要「命令从 PENDING 到 DONE/FAILED」叙事时，用 **事先准备的列表截图或录屏**，避免在观众面前触发新命令。

---

## 2. 时间轴总览（20 分钟）

| 时段 | 长度 | 内容 |
|------|------|------|
| A | 2:00 | 开场、身份介绍、红线与免责声明 |
| B | 3:00 | `/demo` 只读入口页 |
| C | 5:00 | `/ops` 运维与状态 |
| D | 5:00 | `/commands` 命令流（只读浏览） |
| E | 2:00 | `/checklist`、`/recovery`（有则略看，无则口述路线图） |
| F | 2:00 | 只读检查脚本与「系统是否正常」判定概念 |
| G | 1:00 | 收尾、邀请反馈 |

可根据现场提问 **压缩 E/F**，但 **A 段红线不可省**。

---

## 3. A 段（0:00–2:00）开场与红线

**必说内容（建议原句或等价表述）：**

1. 「感谢来参加，今天大约 20 分钟。」
2. 「Project Anchor 当前是 **Early Demo**：只展示 **纪律执行、风控检查、命令流和运维状态**，**不是**自动交易系统，**不是**收益工具，**不接**大家资金，**不接**真实交易所 API key。」
3. 「我们不会在今天这场演示里做任何真实下单，也不会收集大家的密钥。」
4. 「如果大家希望的是『托管交易』或『保证赚钱』，我们可以直接说明：**当前产品边界不包含这些**。」

**必不说：** 盈利承诺、无风险、已可生产接资金、演示即代表可托管（参见 Scope §4）。

---

## 4. B 段（2:00–5:00）`/demo` 只读入口

1. 打开 **`/demo`**（或从首页点「Early Demo」）。
2. 指读 **定位段落**：帮助执行纪律、检查风险、减少情绪化操作；**当前不执行真实交易**。
3. 指读 **黄色提示条**：不接资金、不接 key、不承诺收益。
4. 快速过一遍 **四个链接**：`/ops`、`/commands`、`/checklist`、`/recovery`——说明这是「能力地图」，checklist/recovery 若 404 说明仍在路线图。
5. 点 **「复制只读检查命令说明」**，说明：运维侧可用脚本得到 `SYSTEM_NORMAL / SYSTEM_NOT_NORMAL`，**本页不执行**脚本。

**过渡句：** 「下面我们看两块核心屏：先看系统运维状态，再看命令是怎么被排队和执行的。」

---

## 5. C 段（5:00–10:00）`/ops`

1. 打开 **`/ops`**。
2. 解释 **目的**：在不确定策略盈亏时，先确认 **系统是否健康、开关与工人是否在动**。
3. 结合界面逐项 **只读讲解**（字段名以实际 UI 为准）：  
   - kill switch / 紧急边界（若展示）  
   - worker 是否在处理、有无明显 panic 或长时间停摆  
   - 近期事件或摘要（若有）
4. 强调：**这是观察与治理视角**，不是「帮我下单」。

**若页面报错或空白：** 切到备案截图，并口头说明「演示环境波动，生产上会先修 ops 再看策略」。

---

## 6. D 段（10:00–15:00）`/commands`

1. 打开 **`/commands`**（或仅展示预录屏/截图）。
2. 解释 **命令流模型**：指令进入系统 → **PENDING / RUNNING → DONE 或 FAILED**；失败可重试与否取决于命令类型（可举 **模拟** 例子，不引用真实账户）。
3. **对外场子：** 表格 **上下滚动**，点开 **历史命令详情** 若仅为只读则可；**不要** 演示 Create、Retry。
4. 连接交易者语境：「对应到大家日常，就是：**先定义动作，再看系统是否按纪律执行完毕**，而不是盯盘手搓。」

**金句（可选用）：** 「我们演示的是 **执行与可观测性**，不是本期盈亏。」

---

## 7. E 段（15:00–17:00）`/checklist` 与 `/recovery`

1. 依次点击 **`/checklist`**、**`/recovery`**（若 404，停留 10 秒说明「控制台路由未全量挂载，路线图已有」）。
2. 口述：**checklist** = 负责人在关键步骤上的检查节奏；**recovery** = 出问题时的恢复边界与顺序（以文档与 UI 为准）。
3. 不承诺「已覆盖所有交易所故障场景」。

---

## 8. F 段（17:00–19:00）只读检查与「是否正常」

1. 回到 **`/demo`** 或口头总结，不提具体内网 IP。
2. 说明：仓库内有 **`cloud_runtime_check_readonly.sh`**，在主机上跑可汇总 compose、端口、`/health`、`/ops/state`、worker 日志等，输出 **Final: SYSTEM_NORMAL / SYSTEM_NOT_NORMAL**。
3. 对齐 Scope：**端口应收口到 127.0.0.1**、`/health` 可为 `{"ok":true}`——这是 **运维证据**，不是「策略有效」证据。

---

## 9. G 段（19:00–20:00）收尾与反馈

1. 重复 **对外一句话版本**（见 Scope §5）。
2. 邀请 **2–3 个具体问题**（示例）：  
   - 「哪里最像你现在交易里缺的一环？」  
   - 「哪里你觉得像玩具、不可信？」  
   - 「若永远不做真实下单，这套东西对你还有价值吗？」
3. 致谢；说明后续若合作，仍以 **文档边界 + 法务/合规** 为准。
